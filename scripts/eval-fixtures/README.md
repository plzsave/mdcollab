# eval フィクスチャ（意図的に欠陥を仕込んだ検証用 MD）

`scripts/eval-anchor.ts` / `scripts/eval-review.ts` が読み込む**検証専用**のドキュメント。
「GitHub 連携強化（MD を Issue 化）」というありそうな計画書の体裁で、**わざと問題を埋め込んで**ある。

⚠ これは実在の計画ではない（mdcollab に Issue 化機能の予定はない）。本物の設計書 `docs/` とは別物。
意図的な誤りを含むので **本番ドキュメントとして参照しないこと**。

## 使い方

**1) アンカー当たり率**（逐語引用→本文スパン対応づけ）— このフォルダの *.md を自動で読む:

```bash
EVAL_MODEL=claude-opus-4-8 ANTHROPIC_API_KEY=sk-... bun run eval:anchor
```

**2) 指摘力**（仕込んだ問題を見つけられるか）— いちばん効くのは**アプリの AI レビューに貼って、
「リポジトリ参照を含める」を ON** にする方法。`review-repo` がこの mdcollab のコードを `fetch_repo_file`
等で読み、★ のようなコードとの矛盾（例: 「平文保存」は実装の暗号化と矛盾）まで突いてくる。
（`bun run eval:review` は別の組み込みケース＋インジェクション canary を回すもので、このフォルダの
フィクスチャは読まない。）

下が「期待される指摘（＝仕込んだ問題）」。レビュー出力の答え合わせに使う。

## github-issue-from-md.md の仕込み

- ★ **事実誤り＋セキュリティ**: 「PAT は `ai_keys` テーブルに**平文で保存**」→ 実装は暗号化保存
  （`ai_keys.encrypted_key` / `decryptSecret`）。逆の記述。
- **API 誤り**: 「Issue の作成は `GET /repos/{owner}/{repo}/issues`」→ 作成は **POST**。
- **内部矛盾**: 「必要な PAT 権限は `repo:read` のみで十分」↔ 後段「Issue を**作成・編集**する」
  （書き込みには write 権限が必要）。
- **曖昧**: 「同期のタイミングは**適切に判断**して行う」「内容を確認する（誰が・何を）」。
- **壊れたリンク**: 「[sync-design](./github-sync-design.md)」→ 実在しない。
- **用語ゆれ**: Issue / issue / 課題、ドキュメント / 文書 が混在。
- **セキュリティ欠落**: 「Issue 作成に**レート制限は設けない**」。
- **未定義語**: 「idempotency キー」「GHES」の定義・手段が無い。

## github-sync-api-spec.md の仕込み（構文密度が高くアンカーが難しい側）

- **命名不一致**: リクエストは `doc_id`、レスポンスは `documentId`。
- **型不一致**: `version` がレスポンス例で文字列 `"1"`（版番号は整数のはず）。
- **誤ったステータス**: 表で `404 | UNAUTHORIZED`（認証失敗は **401**）。
- **曖昧**: 「失敗時は**適宜リトライ**する」「本文が大きい場合は**適切に切り詰める**」。
